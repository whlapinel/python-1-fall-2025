Lesson 1.9: Project Work Session Activity Guide

Opening (5 minutes)

Objectives Review and Status Check

  • Quick review of PCEP Section 1 objectives

Detailed Activity Instructions

Activity 1: Project Status Triage (5 minutes)

Purpose: Assess where students are and group support accordingly

Instructions:

  1. Display status categories on board

  2. Have students self-identify their current status:

    • ‚úÖ Complete: Done and tested

  3. Note numbers in each category for targeted support

Activity 2: Rapid Debugging Workshop (10 minutes)

Purpose: Address common errors before students encounter them

Common Error Examples (display and fix together):

# Error 1: Type mismatch
age = input("Enter age: ")
future_age = age + 10  # TypeError!

# Error 2: Operator precedence confusion  
result = 2 + 3 * 4 ** 2  # What's the result?

# Error 3: String formatting issues
name = "Alice"
score = 95.7
print("Student: " + name + " Score: " + score)  # TypeError!

# Error 4: Variable naming
2numbers = 10  # SyntaxError!
my-variable = 5  # SyntaxError!

Student Task: Identify and explain each error, then provide the fix

Activity 3: Partner Code Testing (5 minutes)

Purpose: Catch bugs through fresh eyes

Instructions:

  1. Pair students with someone at similar progress level
  2. Each partner runs the other’s code with these test cases:
    • Normal expected input
    • Zero or negative numbers (where applicable)
    • Very large numbers
    • Text when numbers expected
  3. Provide feedback on user experience and clarity
  4. Return to own projects with insights

Instructor Circulation Strategy

Round 1 (Minutes 5-20): Triage Support

Priority Order:

  1. ‚ùì Confused students: Help clarify requirements and get started
  2. üÜò Help needed: Address specific technical issues
  3. üìà Progress students: Quick check-ins, anticipate next questions

Round 2 (Minutes 25-40): Quality Assurance

Focus Areas:

  1. Code quality: PEP-8 compliance, readability
  2. PCEP coverage: Ensure all objectives met
  3. Testing: Help students test edge cases
  4. Polish: Suggest improvements for early finishers

Common Questions and Suggested Responses

“My code doesn’t work!”

Response Strategy:

  1. “What error message are you seeing?”
  2. “Walk me through what you expected vs. what happened”
  3. “Let’s look at line X together”
  4. Guide them to find the solution rather than fixing it for them

“Is this enough to meet the requirements?”

Response Strategy:

  1. Review their code against the PCEP checklist
  2. “Show me where you demonstrate [specific objective]”
  3. Suggest specific additions if objectives are missing
  4. Praise what they’ve done well

“I’m done early, what should I add?”

Enhancement Suggestions:

  • Input validation (reject invalid entries gracefully)
  • More sophisticated error handling
  • Additional numeral system conversions
  • Menu system for multiple operations
  • Better output formatting
  • More interactive features

“I don’t think I’ll finish in time”

Support Strategy:

  1. Help prioritize core requirements over polish
  2. “What’s the minimum viable version that meets all PCEP objectives?”
  3. Offer to help troubleshoot the biggest blocker
  4. Remind about office hours for additional support

Assessment Notes for Instructor

What to Look For:

  • Functionality: Does the program run without errors?
  • PCEP Coverage: Are all Section 1 objectives clearly demonstrated?
  • Code Quality: Professional presentation, PEP-8 compliance
  • User Experience: Clear instructions, good formatting
  • Understanding: Code shows comprehension, not just copying

Red Flags:

  • Copy-pasted code from internet (unusual complexity for beginner)
  • Missing major PCEP objectives
  • Poor variable naming or no comments
  • Program crashes on normal input
  • No evidence of testing

Quick Feedback Phrases:

  • “Great use of [specific PCEP objective]!”
  • “Consider adding a comment here to explain why…”
  • “This variable name could be more descriptive”
  • “Nice job handling the user input conversion!”
  • “Remember to test with different types of input”
